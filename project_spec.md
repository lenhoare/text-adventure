# LLM Text Adventure Engine — Project Spec

## 1. Purpose

Build a small, SQLite-backed interactive fiction engine where a human and an LLM can both play and create the world during play.

The engine keeps canonical truth: location, inventory, rooms, exits, flags, items, drafts, saves, event logs, and transcripts. The LLM acts as narrator, parser, authoring assistant, and improvisational generator, but it does not directly mutate canonical state. It proposes structured JSON patches which the engine validates and commits.

The first version should be deliberately small, local-first, and hackable.

---

## 2. Core Principles

1. **Graph-first world model**  
   The world is a graph of meaningful places connected by exits/actions, not primarily a coordinate grid.

2. **SQLite as the durable store**  
   SQLite stores worlds, rooms, items, saves, events, drafts, transcripts, and JSON payloads.

3. **JSON as the interchange format**  
   Rooms, items, actions, rules, drafts, patches, and saves are represented as schema-valid JSON.

4. **LLM proposes, engine disposes**  
   The LLM may narrate freely, but canonical changes happen only via validated patches.

5. **Play and authoring are interleaved**  
   A room can begin as narration, become a draft, then be committed into the world.

6. **Randomness is external to the LLM**  
   Dice rolls, card draws, random table results, and seeds are generated by the engine and logged.

7. **Everything important is replayable**  
   Event logs are append-only. Saves are snapshots plus event history pointers.

## 3. Main User Experience

The user can play naturally:

```text
> go north
> examine the brass key
> climb the tree
> talk to the woman
```

The LLM interprets ambiguous prose into candidate actions. The engine validates the action against the current state. The engine applies any legal state transition. The LLM narrates the result.

The user can author naturally:

```text
> make this room real
> commit that description west of the hall
> add a locked pantry north of the kitchen
> generate five hidden rooms under the cellar, but don't spoil them
```

The LLM converts the request into a patch. The engine validates and stores it as a draft or commits it directly depending on user intent.

## 4. Important Concepts

### 4.1 Committed World

The committed world is canonical. It contains rooms, items, NPCs, exits, rules, flags, and authored content that definitely exists.

### 4.2 Draft World

Drafts are structured proposals. They may come from LLM narration, user description, or batch generation.

Drafts can be revised repeatedly before commit.

### 4.3 Provisional Narration

The LLM may imply possible places or objects in prose. These are not real until transformed into a draft or committed patch.

### 4.4 Blank Exits

A committed room may contain an exit whose target is not yet generated.

Example: "through the leaning gate" may be represented as a blank movement action. When the player takes it, the LLM can propose the destination room.

### 4.5 Hidden Content

The world may contain committed but undiscovered rooms. These exist in the database but should not be shown to the player until revealed by play.

---

## 5. Room Lifecycle

A room can move through these states:

```text
narrated -> drafted -> committed -> discovered -> visited
```

Suggested statuses:

| Status | Meaning |
|---|---|
| `proposed` | Exists only as draft JSON. |
| `committed` | Canonical and stored in the world graph. |
| `hidden` | Canonical but not revealed to player. |
| `discovered` | Known to the player but not necessarily visited. |
| `deprecated` | Kept for history but no longer active. |

---

## 6. World Model

A world is a named collection of rooms, items, NPCs, rules, tables, decks, and saves.

### 6.1 Room

A room should include:

- `id`
- `name`
- `description`
- `region`
- `status`
- `exits`
- `items`
- `npcs`
- `tags`
- optional `coords`
- optional `metadata`

Coordinates are metadata only. The graph is authoritative.

### 6.2 Exit / Movement Action

Movement actions connect rooms.

An exit should include:

- `id`
- `kind: "movement"`
- `label`
- `aliases`
- `target`
- `status`
- optional `requires`
- optional `effects`
- optional `generator_hint`
- optional `return_action`

Targets may be:

- existing room id
- draft room id
- `null` for blank exits

### 6.3 Interaction Action

Interaction actions affect state without necessarily moving the player.

Examples:

- examine mirror
- pull lever
- unlock pantry
- talk to ferryman
- listen at door

### 6.4 Items

Items can be portable, fixed, hidden, locked, containers, keys, clues, tools, or scenery.

### 6.5 NPCs

NPC support can be minimal in v1. NPCs are objects with name, description, location, flags, and dialogue hooks.

---

## 7. Engine Responsibilities

The engine must own:

- canonical state
- validation
- persistence
- random number generation
- event log
- save/load
- action application
- draft commit/reject/revise

The engine must not rely on the LLM for:

- current location
- inventory contents
- whether an exit exists
- whether an item has been taken
- dice/card results
- save/load correctness

---

## 8. LLM Responsibilities

The LLM may:

- narrate room descriptions
- parse user input into candidate actions
- propose patches
- generate draft rooms/items/NPCs
- revise drafts from user feedback
- summarize transcripts
- suggest available actions in natural language

The LLM must return structured JSON when asked for mutations.

---

## 9. Patch Model

World changes should be represented as patches.

Initial patch operations:

```text
add_room
update_room
add_exit
update_exit
add_item
move_item
set_flag
clear_flag
add_draft
commit_draft
reject_draft
batch_add_region
```

Every patch should include:

- operation type
- payload
- source turn
- author: `user`, `llm`, or `system`
- validation result
- commit status

---

## 10. SQLite Storage Model

SQLite should store canonical entities and JSON payloads.

Use relational columns for frequently queried fields, and JSON columns for flexible structured data.

Recommended tables:

- `worlds`
- `rooms`
- `items`
- `npcs`
- `actions`
- `drafts`
- `saves`
- `events`
- `transcripts`
- `random_events`

Use foreign keys. Enable WAL mode.

---

## 11. Save/Load

A save should include:

- world id
- current location
- inventory
- flags
- stats
- RNG state or seed
- turn number
- pointer to last event id
- optional state snapshot JSON

The game can be restored from either:

1. save snapshot, or
2. event replay from world start.

For v1, snapshot restore is enough. Event replay can come later.

---

## 12. Randomness

Randomness must be generated by the engine and logged.

Supported v1 forms:

- `1d6`
- `2d10+3`
- coin flip
- draw from named deck
- choose from weighted table

Each random event should store:

- expression/type
- result
- seed or RNG state reference
- turn number
- reason

---

## 13. Minimal Command Set

Internal commands:

```text
/play <input>
/draft room from last narration
/draft room from <description>
/revise draft <id> <instruction>
/commit draft <id>
/reject draft <id>
/generate region <instruction>
/show room
/show map
/save <name>
/load <name>
/export world
```

Natural language can map onto these commands.

---

## 14. Example Flow: Commit Narrated Room

1. User tries to go west.
2. Engine sees no committed west exit.
3. LLM narrates a possible passage behind a velvet curtain.
4. User says: "I like that. Commit it."
5. LLM proposes `add_room` and `add_exit` patch.
6. Engine validates:
   - room id unique
   - origin room exists
   - exit label present
   - return path exists unless marked one-way
7. Engine commits patch.
8. Event is logged.
9. Save can persist the result.

---

## 15. Validation Rules for v1

A committed room must have:

- unique id
- name
- description
- status
- region

A movement action must have:

- unique id within source room
- label
- aliases
- valid source room
- target room id, draft id, or null blank target

A committed two-way exit should have a return path unless explicitly marked one-way.

An item must have:

- unique id
- name
- description
- portability flag

A save must have:

- valid world id
- valid current room id
- inventory containing valid item ids

## 18. Non-goals for v1

Avoid initially:

- full natural-language parser
- combat system
- complex physics
- multiplayer
- web hosting
- procedural infinite worlds
- complex NPC memory
- visual map editor

The point of v1 is to prove the loop:

```text
play -> narrate -> draft -> validate -> commit -> save -> resume
```

---

## 19. Design Motto

The LLM can dream. The engine remembers.
